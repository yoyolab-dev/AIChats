# ADR-001: 选择 Fastify 作为 HTTP 框架

## 状态
✅ 已采纳

## 上下文

我们需要为 AIChats 后端选择一个高效、可扩展的 HTTP 服务器框架。项目使用 Node.js，需要支持插件化、良好的性能以及类型安全（TypeScript 友好）。

## 考虑的方案

- **Express**: 最流行的 Node.js 框架，生态系统庞大，但性能一般，插件机制不如 Fastify 完善。
- **Koa**: 轻量、现代，但中间件生态相对较少，需要自行集成很多功能。
- **Fastify**: 高性能，内置插件架构，强类型支持，与 Prisma 等 ORM 兼容良好。

## 决策

采用 **Fastify** 作为后端 HTTP 框架。

### 理由

1. **性能**: Fastify 在基准测试中表现优异，能够支撑高并发场景。
2. **插件化**: Fastify 的插件系统使得编译、验证、错误处理等功能模块清晰。
3. **开发体验**: 与 TypeScript 集成良好，路由定义简洁。
4. **开箱即用的功能**: 自带 CORS、Helmet 等安全插件，易于配置。

## 后果

- 团队需要学习 Fastify 的插件和 hooks 机制。
- 生态系统虽不如 Express 庞大，但常用需求均有成熟插件。
- 应用代码结构将围绕插件和路由模块组织（`src/routes/*`, `src/plugins/*`）。
