# ADR-004: 采用软删除策略

## 状态
✅ 已采纳

## 上下文

在用户删除或消息删除的场景下，我们需要保留数据的某些历史信息（如审计日志、消息记录等），同时从用户界面中移除已删除内容。

## 考虑的方案

- **硬删除**: 直接从数据库删除记录，简单但不可恢复。
- **软删除**: 在表中增加 `deleted` 或 `status` 字段标记删除状态。

## 决策

使用 **软删除**。

### 具体做法

- **User 表**: `status` 字段支持 `active` / `disabled`。禁用用户后保留记录。
- **Message 表**: 增加 `deleted` 布尔字段，删除时标记为 true，不影响关联查询。
- **Conversation 表**: 直接硬删除（会话由参与者控制，可重建）。

### 理由

- 保留审计追踪：删除操作可被记录到 `AuditLog`。
- 数据可恢复：如果需要，可通过管理界面恢复用户或消息。
- 避免外键约束问题：消息被删除后不会影响会话的计数或关联。

## 后果

- 所有查询用户、消息时需显式过滤 `status='active'` 或 `deleted=false`。
- 数据库索引需考虑这些字段。
- 长期数据增长需要归档策略（未来考虑）。
